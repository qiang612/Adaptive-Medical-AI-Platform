| **1.1 选题背景与应用价值**  **题目来源：**工程实际  **工程背景**：  随着人工智能技术的发展，医疗AI在特定病种的诊断上已取得显著成果。然而，现有的医疗辅助系统多为“烟囱式”架构，即针对单一病种开发独立软件，导致医院需要维护多套系统，数据难以互通。更关键的问题在于，不同疾病的诊断模态差异巨大：有的仅需医学影像，有的则需要影像结合电子病历和生化指标[1]。目前的通用平台难以适应这种异构性，往往需要针对新模型重新开发前端界面和后端接口，扩展成本极高。此外，随着医学大模型和复杂深度学习模型的引入，推理耗时增加，传统的同步请求方式容易导致系统阻塞。因此，开发一套具备UI自适应渲染能力且内置高性能任务调度框架的通用医疗推理平台，成为解决医疗AI落地“最后一公里”的关键。  **应用价值：**  极低的接入成本： 平台通过定义统一的模型描述协议，能够自适应地为单模态或多模态模型生成前端交互界面。接入新病种时，无需修改代码，仅需配置参数即可上线。  高并发与稳定性： 引入异步任务调度机制，将耗时的模型推理与Web服务解耦，确保在多人同时使用或模型计算量大时，系统依然流畅稳定。  打破医疗数据孤岛：构建统一的多模态数据处理管道，能够同时处理文本病历和影像数据，为医生提供更全面的患者画像，而非单一维度的检查结果。  科研与教学平台化：本系统支持模型的动态集成与对比，可作为科研人员验证新算法的实验平台，也可作为医学生学习典型病例的教学工具。  降低基层医疗门槛：将专家级的诊断模型集成到Web端，有助于基层医疗机构通过互联网获取高质量的辅助诊断服务，促进医疗资源均衡化。  **理论意义**：  本课题深入探索了自适应软件架构在医疗AI通用平台中的应用机理。通过研究基于元数据驱动的前端UI动态渲染机制与异构模型统一适配协议，从理论上解决了传统医疗辅助诊断系统前端界面与底层算法模型高度耦合、难以通用的技术难题。同时，本课题将分布式异步任务调度理论应用于医疗推理场景，构建了计算密集型AI任务与Web交互服务解耦的高性能处理架构[2]。研究成果为 “互联网 + 医疗健康” 领域提供了可复用的平台架构参考，对推动智慧医院建设中 AI 技术的规模化应用具有重要的理论与实践价值。 |
| **1.2 国内外研究现状分析**  **国内现状：**我国在医疗AI领域发展迅速，腾讯觅影、百度灵医等企业已推出成熟的影像辅助诊断产品，主要集中在肺结节、眼底病变筛查等领域。在学术界，研究多集中于提升特定算法的准确率[3]。然而，在软件系统层面，现有的医院信息系统多侧重于行政流程管理，缺乏对AI模型的深度集成能力。大多数辅助诊断系统仍是针对单一病种开发的硬编码软件，缺乏通用性和可扩展性，难以在一个平台上同时管理多种疾病模型。  **国外现状：**国外以 IBM Watson Health、Google DeepMind 为代表，在多模态医疗数据处理方面处于领先地位[4]，开源社区如MONAI也提供了丰富的模型资源[5]。在系统架构方面，MaaS模式已成为主流，但现有研究多聚焦于后端推理精度[6]，忽略了应用层系统的灵活性，普遍存在前端界面与特定模型高度耦合、同步推理机制易阻塞系统等工程弊端，缺乏能够根据模型特性自适应生成交互界面、并支持大规模异步任务调度的通用医疗推理平台。 |

| **2.1 研究内容与目标**  **研究内容**：  1.自适应平台架构设计：  设计模型元数据协议：定义一套JSON规范，用于描述模型的输入类型、参数范围及输出格式。  后端自适应解析器：开发通用的模型加载与解析模块，根据Schema动态校验前端传输的数据，并自动路由到对应的预处理管道。  前端自适应渲染引擎：基于Vue 3的动态组件，实现“配置驱动UI”。例如，当加载冠心病模型（多模态）时，界面自动渲染出“CT图片上传框”+“血压/血糖输入框”；当加载宫颈癌模型（单模态）时，仅显示“CT图像上传框”。[7]  2.基于任务调度的异步推理框架：  引入 Celery 作为分布式任务队列，Redis 作为消息中间件。[8]  设计生产者-消费者模型：Web后端作为生产者将诊断请求封装为Task推入队列；AI推理服务作为消费者从队列获取任务执行。  实现状态轮询与通知：前端通过轮询或WebSocket实时获取任务状态（排队中 -> 处理中 -> 完成/失败）。  3.多模态医疗模型的集成与适配：  模型准备：选取具有代表性的开源模型或根据公开数据集进行轻量化训练，准确率达70-80%即可。  单模态模型：集成基于YOLOv8的宫颈癌检测模型。[9]  多模态模型：集成一个基于CNN + MLP混合架构处理“图像+临床表格数据”的冠心病风险预测模型。[10]  多模态适配器：编写Python适配器类，将平台传入的标准数据流转换为PyTorch/Scikit-learn模型所需的Tensor或DataFrame格式，解决多模态数据的对齐与融合问题。  4.可视化与结果反馈：  根据模型输出类型，自适应展示结果。如果是分类任务，显示概率仪表盘；如果是检测任务，在原图上绘制Bounding Box；如果是多模态预测，展示关键特征权重。  **研究目标**：  1. 完成平台的开发，实现“一个平台，运行多种模型”的目标。  2. 成功集成至少一个单模态模型和一个多模态模型，验证系统的自适应能力。  3. 通过任务调度框架，实现系统在多任务并发下的稳定性。  4. 通过软件工程方法，规范化系统的需求分析、设计、编码及测试流程。  **技术指标：**  响应速度：单次诊断推理及页面渲染时间 < 3秒。  模型精度：宫颈癌分类准确率 > 80%，冠心病预测AUC值 > 0.75。  并发性能：系统支持至少50人同时在线访问不卡顿。  兼容性：支持JPG/PNG/DICOM格式图片上传及Excel数据批量导入。 |
| **2.2 拟采取的研究方案**  **需求分析**：  本系统旨在构建一个通用的、可配置的AI医疗推理平台，解决传统医疗软件针对单一病种硬编码开发、扩展性差的问题。通过调研临床实际与软件工程最佳实践，确定系统需具备“前端UI自适应渲染”、“多模态数据动态融合”及“异步任务调度”三大核心能力[11]。  1. 医生端需求：  自适应交互界面： 系统需废除固定表单模式。当医生选择不同病种时，界面应根据后台配置动态生成对应的输入组件。  异步诊断与任务追踪： 针对医学大模型推理耗时较长的特点，系统需支持“提交即走”模式。医生提交病例后无需在页面阻塞等待，系统应提供“任务列表”实时显示排队状态和完成状态，并在推理完成后发送通知。  多模态结果可视化： 诊断结果需支持多形式展示。  影像类：需支持在原始DICOM/JPG图片上通过Canvas图层叠加病灶标注框或热力图。  数值/文本类：需通过仪表盘展示患病概率，并列出高风险因子权重，实现结果的“可解释性”。  2. 管理员端需求：  模型注册与Schema配置： 管理员不仅是上传模型权重文件，更需定义模型的输入/输出协议。  配置内容：需指定模型需要什么类型的数据、字段名称以及前端组件的渲染类型  服务状态监控： 需具备对任务调度队列的可视化监控能力，实时查看当前积压的任务数量、Worker节点的健康状态以及推理失败的错误日志，以便及时处理死锁或服务宕机问题。  用户权限管理： 基于RBAC模型，严格控制不同科室医生对特定疾病模型的访问权限。  3. 非功能性需求：  高并发与任务调度： 系统必须将Web服务与AI推理服务解耦。需引入消息队列和任务分发机制，确保在多用户并发提交高清影像分析时，主系统界面不卡顿、不崩溃。  兼容性与扩展性： 平台需通过适配器模式屏蔽底层算法差异，能够同时兼容基于PyTorch的深度学习模型和基于Scikit-learn的机器学习模型，实现异构模型的“即插即用”。  响应性能： 界面动态渲染时间应 < 0.5秒；在任务排队机制下，需明确告知用户预计等待时间。  **软硬件开发平台参数：**  1.前端开发：Vue 3, Element Plus, Axios, JSON Schema Form  2.后端开发：Python 3.9, FastAPI , SQLAlchemy  3.AI推理环境：PyTorch, Scikit-learn, OpenCV  4.数据库：MySQL 8.0  5.任务调度：分布式任务队列Celery, 消息中间件Redis  6.开发环境：PyCharm / VS Code / CUDA加速环境/Google colab云端训练  **技术路线：**  1. 模型自描述协议与数据库设计：  元数据协议定义：设计一套基于JSON Schema的标准模型描述协议 。该协议用于定义任意AI模型的输入参数和输出格式 ，是实现系统“自适应”的基石。  数据模型构建：设计支持动态扩展的数据库Schema。除了基础的User和Record表外，重点设计ModelRegistry和TaskQueue，实现模型配置与业务数据的解耦。  2. 高性能异步任务调度架构搭建：  生产者-消费者模型实现： 引入 Redis 作为消息中间件，Celery 作为分布式任务执行框架。  调度逻辑开发：后端服务FastAPI作为生产者，负责接收HTTP请求并生成Task ID推入队列；独立运行的Worker进程作为消费者，负责从队列获取任务并加载AI模型进行重度计算。  状态同步机制：实现基于轮询或 WebSocket 的任务状态查询接口，确保前端能实时获取“排队中”、“处理中”或“已完成”的任务状态。  3. 前端自适应渲染引擎开发：  配置驱动UI：基于 Vue 3 开发通用的动态表单生成器。实现递归解析后端返回的 JSON Schema，将数据类型自动映射为对应的 Element Plus UI 组件。  动态可视化适配：开发可复用的结果展示组件。针对分类任务渲染仪表盘，针对分割任务渲染 Canvas 叠加层，实现“一套代码，适配多种模型结果”。  4. 多模态模型适配与集成：  适配器模式应用：定义标准的 Python 抽象基类，开发通用模型适配器。无论是 PyTorch 的图像模型还是 Scikit-learn 的数值模型，均通过适配器封装为统一的 .predict() 接口。  预处理管道标准化：集成OpenCV 和 Pandas，构建可配置的数据预处理流水线，自动处理图像的 Resize/Normalization 和表格数据的缺失值填充，确保异构数据能被系统正确识别[12]。  5. 系统全链路测试与优化：  并发压力测试： 使用 JMeter 模拟多用户同时上传高清影像，验证 Celery 任务队列的削峰填谷能力及系统的稳定性。  自适应功能验证： 在不修改前端代码的前提下，热加载一个新的测试模型，验证界面是否能正确生成对应的输入表单和结果视图。 |
| **2.3 预期成果与创新性**  成果形式：  一套完整的Web端自适应辅助医疗推理系统源代码。  毕业设计论文及相关开发文档。  训练好的疾病诊断模型权重文件。  代码量：预计 6000+ 行（含前端、后端及模型训练或微调脚本）。  创新性：  1. 基于元数据驱动的自适应UI架构：提出了基于元数据驱动的动态前端渲染机制。摒弃了传统医疗软件针对单一病种硬编码开发界面的模式，通过设计标准化的模型描述协议，实现了UI组件与底层数据模型的动态映射与自动生成。这一架构极大降低了新模型接入的开发成本，显著提升了系统的可扩展性与复用性。  2. 计算与服务解耦的异步任务调度体系：构建了高并发场景下的异步推理调度框架。创新性地引入分布式消息队列，采用生产者-消费者模型将轻量级Web服务与计算密集型AI推理服务完全解耦。有效解决了医学大模型推理导致的线程阻塞问题，实现了系统资源的削峰填谷与任务的流水线处理，保障了多用户访问下的系统稳定性。  3. 异构医疗数据的标准化适配与融合机制：设计了通用的多模态数据适配层。针对医疗场景中非结构化影像与结构化临床文本数据并存的痛点，通过定义统一的输入输出接口协议，在工程层面实现了异构数据的自动对齐与格式标准化。这为多模态AI模型在通用平台上的“即插即用”提供了标准化的工程范式。 |
